\section{Introducción}

La presente práctica tiene como finalidad estudiar el rendimiento computacional de distintas operaciones de álgebra lineal, comparando implementaciones realizadas en GNU Octave, lenguaje C y funciones de la biblioteca CBLAS.

Las operaciones evaluadas fueron el producto punto, la multiplicación matriz-vector y la multiplicación matriz-matriz. Estas operaciones fueron seleccionadas debido a que representan distintos órdenes de complejidad algorítmica: lineal, cuadrático y cúbico, respectivamente.

El trabajo se desarrolló en un entorno WSL, utilizando herramientas de compilación y medición como GCC, Octave, CBLAS, Bash y Gnuplot. Se implementaron versiones manuales en C, versiones optimizadas y se compararon con rutinas especializadas de CBLAS.

El objetivo central fue observar cómo influyen la complejidad algorítmica, el tipo de lenguaje, la optimización de código y la localidad de memoria sobre los tiempos de ejecución.

\subsection{Objetivos específicos}

    \item Implementar y medir el tiempo de ejecución del producto punto, multiplicación matriz-vector y multiplicación matriz-matriz.

    \item Aplicar técnicas de optimización como loop unrolling y cambio de orden de ciclos.

\section{Marco teórico}

\subsection{Complejidad algorítmica}

La complejidad algorítmica permite estudiar cómo crece el costo computacional de un algoritmo a medida que aumenta el tamaño del problema. En esta práctica se analizaron tres operaciones fundamentales.

\subsubsection{Producto punto}

El producto punto entre dos vectores de tamaño $n$ se define como:

\begin{equation}
    c = \sum_{i=1}^{n} x_i y_i
\end{equation}

La operación requiere recorrer los vectores una vez, por lo que su complejidad es:

\begin{equation}
    O(n)
\end{equation}

\subsubsection{Multiplicación matriz-vector}

La multiplicación entre una matriz $A \in \mathbb{R}^{n \times n}$ y un vector $x \in \mathbb{R}^{n}$ se define como:

\begin{equation}
    y_i = \sum_{j=1}^{n} A_{ij}x_j
\end{equation}

Esta operación requiere dos ciclos anidados, por lo que su complejidad es:

\begin{equation}
    O(n^2)
\end{equation}

\subsubsection{Multiplicación matriz-matriz}

La multiplicación entre dos matrices cuadradas $A$ y $B$ de tamaño $n \times n$ se define como:

\begin{equation}
    C_{ij} = \sum_{k=1}^{n} A_{ik}B_{kj}
\end{equation}

Esta operación requiere tres ciclos anidados, por lo que su complejidad es:

\begin{equation}
    O(n^3)
\end{equation}

\subsection{GNU Octave}

GNU Octave permite resolver operaciones de álgebra lineal mediante operadores nativos. Estos operadores suelen estar asociados a rutinas compiladas y optimizadas, por lo que pueden presentar tiempos de ejecución considerablemente menores que una función escrita directamente en Octave mediante ciclos interpretados.

\subsection{CBLAS}

CBLAS corresponde a una interfaz en lenguaje C para acceder a rutinas BLAS. Estas rutinas están diseñadas para operaciones de álgebra lineal y suelen estar altamente optimizadas.

En esta práctica se utilizaron las siguientes funciones:

\begin{itemize}
    \item \texttt{cblas\_ddot}: producto punto.
    \item \texttt{cblas\_dgemv}: multiplicación matriz-vector.
    \item \texttt{cblas\_dgemm}: multiplicación matriz-matriz.
\end{itemize}

\subsection{Optimización de código}

La optimización de código busca reducir el tiempo de ejecución de un programa sin modificar el resultado final. En esta práctica se aplicaron dos estrategias principales:

\begin{itemize}
    \item \textbf{Loop unrolling}: técnica que reduce la sobrecarga de control de ciclos, realizando varias operaciones por iteración.
    \item \textbf{Cambio de orden de ciclos}: técnica orientada a mejorar el acceso a memoria y la localidad de datos, especialmente relevante en operaciones con matrices.
\end{itemize}

\section{Metodología}

\subsection{Entorno de trabajo}

El desarrollo se realizó en WSL utilizando las siguientes herramientas:

\begin{itemize}
    \item GNU Octave para implementar y medir operaciones interpretadas.
    \item GCC para compilar programas en C.
    \item CBLAS mediante la biblioteca \texttt{libblas-dev}.
    \item Bash para automatizar las mediciones.
    \item Gnuplot para generar las gráficas.
    \item LaTeX para elaborar el informe.
\end{itemize}

\subsection{Procedimiento de medición}

Para cada tamaño del problema se ejecutó cada algoritmo siete veces. Luego, se calculó el promedio de los tiempos obtenidos. Esta estrategia permite reducir el efecto de variaciones puntuales del sistema operativo o de la carga de CPU.

En Octave se utilizaron las funciones \texttt{tic} y \texttt{toc}. En C se utilizó la función \texttt{gettimeofday}. Los datos fueron almacenados en archivos \texttt{.dat} y posteriormente graficados con Gnuplot.

\subsection{Tamaños utilizados}

Los tamaños de entrada fueron seleccionados de forma progresiva, usando potencias de dos. Los tamaños de entrada fueron seleccionados experimentalmente hasta encontrar rangos en los cuales tanto Octave como CBLAS entregaran tiempos de ejecución medibles y mayores a 0,01 segundos. Este criterio se utilizó para evitar mediciones poco representativas asociadas a tiempos demasiado pequeños.

\section{Implementaciones}

\subsection{Producto punto}

Para el producto punto se implementaron las siguientes versiones:

\begin{itemize}
    \item Función manual en Octave.
    \item Operador nativo de Octave.
    \item Implementación manual en C.
    \item Implementación optimizada en C mediante loop unrolling.
    \item Función \texttt{cblas\_ddot}.
\end{itemize}

La versión manual en C utiliza un ciclo simple:

\begin{lstlisting}[language=C]
for(i = 0; i < n; i++)
    c += x[i] * y[i];
\end{lstlisting}

La versión optimizada realiza varias operaciones dentro de una misma iteración:

\begin{lstlisting}[language=C]
for(i = 0; i < limit; i += 5)
{
    c += x[i] * y[i]
       + x[i + 1] * y[i + 1]
       + x[i + 2] * y[i + 2]
       + x[i + 3] * y[i + 3]
       + x[i + 4] * y[i + 4];
}
\end{lstlisting}

\subsection{Multiplicación matriz-vector}

La implementación optimizada de la multiplicación matriz-vector se presentó considerando el efecto del orden de almacenamiento en memoria. En lenguaje C los arreglos bidimensionales se almacenan en \textit{row major order}, por lo que el recorrido por filas favorece la localidad espacial y reduce accesos ineficientes a memoria.

La \textbf{versión C row major} (manual) utiliza dos ciclos anidados recorriendo por filas:

\begin{lstlisting}[language=C]
for(i = 0; i < m; i++)
{
    y[i] = 0.0;
    for(j = 0; j < n; j++)
    {
        y[i] += A[i * n + j] * x[j];
    }
}
\end{lstlisting}

La \textbf{versión C column major} (optimizada) reordena los accesos para comparar el efecto de la localidad de datos al recorrer la matriz en distinto orden, haciendo evidente que el rendimiento se ve afectado por el patrón de acceso a memoria.

\subsection{Multiplicación matriz-matriz}

En la multiplicación matriz-matriz se evaluó una \textbf{versión C clásica} y una \textbf{versión C optimizada} mediante cambio de orden de ciclos. Esta reorganización permite recorrer elementos contiguos en memoria, mejorando la localidad de datos y el rendimiento general de la implementación en C.

La implementación manual se realizó mediante tres ciclos anidados en el orden clásico:

\begin{lstlisting}[language=C]
for(i = 0; i < m; i++)
{
    for(j = 0; j < n; j++)
    {
        double sum = 0.0;
        for(p = 0; p < k; p++)
        {
            sum += A[i * k + p] * B[p * n + j];
        }
        C[i * n + j] = sum;
    }
}
\end{lstlisting}

La versión optimizada utiliza el orden de ciclos $i$, $p$, $j$, lo que permite recorrer de forma más contigua la matriz $B$ y la matriz $C$ en memoria:

\begin{lstlisting}[language=C]
for(i = 0; i < m; i++)
{
    for(p = 0; p < k; p++)
    {
        double a_ip = A[i * k + p];
        int rowB = p * n;
        int rowC = i * n;

        for(j = 0; j < n; j++)
        {
            C[rowC + j] += a_ip * B[rowB + j];
        }
    }
}
\end{lstlisting}

\subsection{Relación entre implementación y complejidad algorítmica}

En esta sección se identifica, directamente en el código C, el origen de la complejidad de cada operación.

\subsubsection{Producto punto — $O(n)$}

La complejidad $O(n)$ se observa en la presencia de un único ciclo que recorre los $n$ elementos de los vectores:

\begin{lstlisting}[language=C]
for(i = 0; i < n; i++)
    c += x[i] * y[i];
\end{lstlisting}

\subsubsection{Multiplicación matriz-vector — $O(n^2)$}

La complejidad $O(n^2)$ se aprecia en dos ciclos anidados: uno para las filas de la matriz y otro para las columnas:

\begin{lstlisting}[language=C]
for(i = 0; i < m; i++)
    for(j = 0; j < n; j++)
        y[i] += A[i * n + j] * x[j];
\end{lstlisting}

\subsubsection{Multiplicación matriz-matriz — $O(n^3)$}

La complejidad $O(n^3)$ se identifica en tres ciclos anidados, correspondientes al cálculo de cada elemento de la matriz resultante:

\begin{lstlisting}[language=C]
for(i = 0; i < m; i++)
    for(j = 0; j < n; j++)
        for(p = 0; p < k; p++)
            C[i*n+j] += A[i*k+p] * B[p*n+j];
\end{lstlisting}

\section{Análisis de resultados}

Los resultados muestran que las rutinas especializadas de CBLAS presentan, en general, menores tiempos de ejecución que las implementaciones manuales. Esto se explica por el uso de bibliotecas compiladas y optimizadas para operaciones de álgebra lineal. En el caso de las implementaciones en C, las mejoras obtenidas mediante loop unrolling y reorganización de ciclos evidencian que la optimización de código y la localidad de memoria influyen directamente en el rendimiento.

\subsection{Análisis del producto punto}

El producto punto presentó un crecimiento aproximadamente lineal, coherente con su complejidad $O(n)$. El operador nativo de Octave resultó más rápido que la función manual escrita en Octave, ya que el operador interno está implementado en código compilado. En las implementaciones en C, la versión optimizada mediante loop unrolling redujo el tiempo de ejecución al disminuir la sobrecarga de control del ciclo. A medida que el tamaño del problema aumentó, CBLAS presentó los mejores resultados, confirmando el efecto de su alto nivel de optimización.

\subsection{Análisis de la multiplicación matriz-vector}

La multiplicación matriz-vector presentó un crecimiento superior al producto punto, coherente con su complejidad $O(n^2)$. En esta operación, el orden de almacenamiento en memoria tiene un impacto directo: la versión C row major, al recorrer los datos de forma contigua en memoria, favorece la localidad espacial y reduce accesos ineficientes a caché. La versión CBLAS fue la más eficiente, especialmente en tamaños grandes.

Se observó una medición irregular en $n = 16384$, donde la versión optimizada presentó un tiempo mayor al esperado. Esta irregularidad puede estar asociada a ruido del sistema, carga del procesador o efectos de caché y debe interpretarse como una medición atípica.

\subsection{Análisis de la multiplicación matriz-matriz}

La multiplicación matriz-matriz fue la operación con mayor diferencia entre las implementaciones, resultado esperado dada su complejidad $O(n^3)$. La versión C clásica presentó tiempos muy elevados especialmente para tamaños grandes, por el acceso menos eficiente a memoria. La versión C optimizada mediante cambio de orden de ciclos reorganizó los accesos para recorrer elementos contiguos en memoria, mejorando la localidad de datos: para $n = 4096$, la versión manual tardó 4219.46\,s mientras que la optimizada tardó 22.46\,s.

En comparación, \texttt{cblas\_dgemm} alcanzó 0.386\,s para el mismo tamaño, evidenciando el alto nivel de optimización de BLAS para operaciones matriciales. En cuanto a la comparación Octave operador vs CBLAS, CBLAS superó al operador nativo en los tamaños donde ambas mediciones resultaron representativas.

% =========================
% DISCUSIÓN
% =========================
\section{Discusión}

Los resultados obtenidos permiten observar que la elección de la implementación influye directamente en el rendimiento. Aunque todos los algoritmos resuelven matemáticamente el mismo problema, sus tiempos de ejecución pueden variar de forma considerable.

En Octave, los operadores nativos fueron mucho más eficientes que las funciones implementadas manualmente con ciclos, debido a que las funciones escritas directamente en Octave son interpretadas, mientras que los operadores nativos se apoyan en rutinas compiladas. Sin embargo, CBLAS superó al operador nativo en las operaciones de mayor escala, lo que confirma la ventaja de las bibliotecas especializadas.

En C, las optimizaciones aplicadas permitieron mejorar el rendimiento respecto de las versiones manuales. El loop unrolling redujo la sobrecarga de control en el producto punto. El cambio de orden de ciclos en la multiplicación matriz-matriz y el uso del recorrido row major en la multiplicación matriz-vector tuvieron un impacto significativo al mejorar la localidad de memoria.

Finalmente, las funciones CBLAS demostraron el mejor rendimiento general, ya que BLAS está diseñado específicamente para operaciones de álgebra lineal y aprovecha optimizaciones internas que son difíciles de igualar con implementaciones manuales.

% =========================
% CONCLUSIONES
% =========================
\section{Conclusiones}

Se concluye que la complejidad algorítmica determina el crecimiento general del tiempo de ejecución, pero no explica por sí sola las diferencias de rendimiento entre implementaciones. El producto punto presentó un crecimiento lineal, la multiplicación matriz-vector cuadrático y la multiplicación matriz-matriz cúbico.

Las rutinas CBLAS obtuvieron los mejores resultados en la mayoría de los casos debido a su alto nivel de optimización. Las versiones optimizadas en C mejoraron respecto de las implementaciones manuales, especialmente cuando se aplicaron técnicas como loop unrolling y reorganización del acceso a memoria.

En el producto punto, el loop unrolling redujo la sobrecarga de control del ciclo y mejoró el rendimiento de forma consistente. En la multiplicación matriz-vector, la consideración del almacenamiento row major permitió aprovechar la localidad espacial de los datos. En la multiplicación matriz-matriz, el cambio de orden de ciclos fue la optimización más determinante, reduciendo los tiempos en varios órdenes de magnitud.

En consecuencia, el rendimiento depende tanto del orden de complejidad como de la forma concreta en que el algoritmo interactúa con la arquitectura. La práctica permitió verificar experimentalmente que la optimización de código y la localidad de memoria son factores tan relevantes como la complejidad algorítmica al momento de evaluar el rendimiento computacional real.
